
\chapter{Methodology}

\section{Process Model}
The team is following the Agile process model for the development of TRIBAL. Agile
was chosen because it allows the project to be broken down into small, manageable parts
called sprints, and each sprint produces a working piece of the application. This is useful
for a minor project because requirements can be refined as the team learns more, and
progress can be shown at regular intervals like this mid-term defense.
The project has been divided into the following sprints:
\begin{itemize}
\item Sprint 1: Project setup, requirement analysis, and UI/UX wireframing in Figma.
\item Sprint 2: Flutter onboarding flow, authentication screens (Login and Signup), and
post-signup profile completion screens.
\item Sprint 3: Django backend setup, PostgreSQL database design, and JWT-based
authentication APIs.
\item Sprint 4: Design and implementation of the roommate compatibility scoring
algorithm (current sprint).
\item Sprint 5: Real-time chat system using Django Channels and WebSockets.
\item Sprint 6: Event creation and joining module.
\item Sprint 7: Integration testing between the Flutter frontend and Django backend.
\item Sprint 8: Final testing, bug fixing, and documentation.
\end{itemize}
At the time of this report, the team has completed Sprints 1 to 4 and has started work on
Sprint 5.

\section{Algorithm}

\subsection{Roommate Compatibility Scoring Algorithm}
The Roommate Compatibility Scoring Algorithm is the main algorithm used in the TRIBAL application to recommend compatible roommates. It compares the roommate profiles of two users and calculates a compatibility score between 0 and 100. The algorithm is based on weighted scoring and compares multiple lifestyle preferences such as cleanliness, budget, sleep schedule, smoking, drinking, guests, food preference, study habit, interests, and noise level. Each factor contributes to the final score according to its assigned weight. After calculating the score, the algorithm applies deal-breaker rules for major lifestyle conflicts and ranks the candidates from highest to lowest compatibility.

\subsubsection{Input}
The algorithm takes two roommate profiles as input.
\begin{itemize}
    \item Budget range (Minimum and Maximum)
    \item Sleep schedule
    \item Wake-up time
    \item Cleanliness level
    \item Noise level
    \item Smoking preference
    \item Drinking preference
    \item Guest preference
    \item Food preference
    \item Study habit
    \item Interests
    \item Gender preference
    \item Room type preference
\end{itemize}

\subsubsection{Output}
\begin{itemize}
    \item Compatibility score (0–100)
    \item Score breakdown for each factor
    \item Deal-breaker status (if any)
    \item Ranked list of compatible roommates
\end{itemize}

\subsubsection{Weight Distribution}
\begin{table}[H]
\centering
\begin{tabular}{|l|c|}
\hline
\textbf{Factor} &  \textbf{Weight (\%)} \\
\hline
Cleanliness & 20 \\
Budget & 20 \\
Sleep Schedule & 15 \\
Noise Level & 12 \\
Smoking & 10 \\
Guests Preference & 8 \\
Interests & 6 \\
Study Habit & 4 \\
Food Preference & 3 \\
Drinking Preference & 2 \\
\hline
\textbf{Total} & \textbf{100} \\
\hline
\end{tabular}
\caption{Compatibility Scoring Weights}
\label{table:scoring_weights}
\end{table}
The sum of all weights is 100, which allows the final compatibility score to be calculated directly out of 100.

\subsubsection{Compatibility Score Calculation}
\[
\text{FinalScore} = \sum \text{Score}_i, \quad \text{where } i = 1 \text{ to } n.
\]
Where $\text{Score}_i$ is the score of each matching factor and $n$ is the total number of factors. The final score is limited between 0 and 100.

\subsubsection{Individual Factor Scoring}
\paragraph{Cleanliness}
\[
\text{CleanlinessScore} = 20 \times \left(1 - \frac{|C_1 - C_2|}{4}\right)
\]

\paragraph{Noise Level}
\[
\text{NoiseScore} = 12 \times \left(1 - \frac{|N_1 - N_2|}{4}\right)
\]

\paragraph{Budget Matching}
The algorithm compares the budget ranges of both users. If the ranges overlap, a higher score is assigned. If they do not overlap, the score is reduced according to the difference between the two budget ranges. Maximum BudgetScore = 20.

\paragraph{Sleep Schedule Matching}
\[
\text{SleepCompatibility} = 0.7 \times \text{ScheduleCompatibility} + 0.3 \times \text{WakeTimeSimilarity}
\]
\[
\text{SleepScore} = 15 \times \text{SleepCompatibility}
\]

\paragraph{Smoking Preference}
\[
\text{SmokingScore} = 10 \times \text{CompatibilityValue}
\]
Examples: Non-smoker \& Non-smoker = 1.0, Smoker \& Smoker = 1.0, Non-smoker \& Smoker = 0.0, Non-smoker \& Occasional Smoker = 0.4.

\paragraph{Drinking Preference}
\[
\text{DrinkingScore} = 2 \times \text{CompatibilityValue}
\]

\paragraph{Guests Preference}
Rarely = 0, Sometimes = 1, Frequently = 2.
\[
\text{GuestScore} = 8 \times \left(1 - \frac{|G_1 - G_2|}{2}\right)
\]

\paragraph{Study Habit}
Same study habit = Full score, Different study habit = Half score.
\[
\text{StudyHabitScore} = 4 \times \text{CompatibilityValue}
\]

\paragraph{Food Preference}
Examples: Vegetarian \& Vegetarian = 1.0, Vegetarian \& Vegan = 0.9, Vegetarian \& Non-vegetarian = 0.3.
\[
\text{FoodScore} = 3 \times \text{CompatibilityValue}
\]

\subsubsection{Interest Matching using Jaccard Similarity}
\[
J(A,B) = \frac{|A \cap B|}{|A \cup B|}
\]
\[
\text{InterestScore} = 6 \times J(A,B)
\]
\textbf{Example}: \\
User A: Hiking, Music, Football \\
User B: Football, Music, Gaming \\
Common Interests = 2 \\
Total Unique Interests = 4 \\
$J(A,B) = 2/4 = 0.5$ \\
InterestScore = $6 \times 0.5 = 3$

\subsubsection{Deal Breaker Conditions}
\begin{table}[H]
\centering
\begin{tabular}{|l|c|}
\hline
\textbf{Deal Breaker} &  \textbf{Maximum Score} \\
\hline
Smoking Conflict & 40 \\
Budget Conflict & 35 \\
Cleanliness Conflict & 45 \\
\hline
\end{tabular}
\caption{Deal Breaker Rules}
\label{table:deal_breakers}
\end{table}
Instead of completely rejecting a match, the algorithm limits the maximum compatibility score.

\subsubsection{Candidate Filtering}
Before calculating compatibility scores, the algorithm removes users who do not satisfy the required preferences.
\begin{itemize}
    \item Gender preference
    \item Room type preference
\end{itemize}

\subsubsection{Algorithm Steps}
\begin{algorithm}[!htb]
\caption{Roommate Compatibility Scoring Algorithm}
\label{alg:roommate-matching}
\KwIn{Current user profile and candidate roommate profiles}
\KwOut{Ranked list of compatible roommates}

Get current user's roommate profile\;
Fetch all active roommate profiles\;
Remove the current user from the candidate list\;

\For{each candidate}{
    Check gender preference\;
    Check room type preference\;
    \If{candidate passes the filters}{
        Calculate cleanliness score\;
        Calculate budget score\;
        Calculate sleep schedule score\;
        Calculate noise level score\;
        Calculate smoking score\;
        Calculate drinking score\;
        Calculate guest preference score\;
        Calculate food preference score\;
        Calculate study habit score\;
        Calculate interest score using Jaccard Similarity\;
        Add all scores to obtain the total compatibility score\;
        Apply deal-breaker rules if necessary\;
        Store the candidate and final score\;
    }
}

Sort all candidates in descending order of compatibility score\;
Return the ranked roommate list\;
\end{algorithm}

\subsubsection{Final Ranking}
After calculating the compatibility score for every candidate, the algorithm sorts all candidates in descending order based on their final scores. The users with the highest compatibility scores are displayed first in the roommate recommendation section.

The proposed algorithm is transparent, easy to understand, and computationally efficient. Since it uses weighted scoring instead of machine learning, it does not require training data and can provide explainable roommate recommendations suitable for the current implementation of the TRIBAL application.

Figure~\ref{fig:flowchart} shows how the roommate matching algorithm works step by step. The flowchart starts with a ``Start'' node, followed by the step ``User requests roommate matches''. After that, the system fetches the onboarding data of the requesting user from the database, and then fetches all candidate user profiles. For each candidate, the algorithm calculates the individual factor match values and then applies the factor weights. The next step is calculating the total compatibility score, after which there is a decision diamond labelled ``Deal breaker detected?''. If the answer is Yes, the score is capped at the maximum allowed value. If the answer is No, the calculated score is kept as it is. After this, the final score is rounded and all candidates are sorted by their compatibility score. Finally, the ranked list is returned to the Flutter app and the flowchart ends. This diagram helped us make sure that we had thought through all the edge cases in the algorithm before starting to code it.

    \end{minipage}%

\end{figure}
Figure~\ref{fig:class_diagram} shows the class diagram for the TRIBAL system. The central class is \texttt{User}, which has four attributes: id, username, email, and password. From the \texttt{User} class, five relationships branch out. The \texttt{User} owns a \texttt{Profile} (one-to-one), which holds gender, age, bio, and location. The \texttt{Profile} further has a one-to-one relationship with a \texttt{Preferences} class, which stores the lifestyle fields used in matching: sleepSchedule, cleanliness, budget, and interests. The \texttt{User} also receives \texttt{Match} objects (one-to-many), where each \texttt{Match} holds a compatibilityScore and a matchedAt timestamp. The \texttt{User} creates or joins \texttt{Event} objects, each of which has a title, description, location, and eventDate. Finally, the \texttt{User} participates in a \texttt{ChatRoom}, which contains many \texttt{Message} objects, each with a content string and a timestamp. The \texttt{User} also has a sends relationship to \texttt{Message} directly. This diagram shows us all the data models we need to build and how they relate to each other.



\subsection{Sequence Diagram}
\begin{figure}[H]
    \centering
    \makebox[\textwidth][c]{%
    \begin{minipage}{1.9\textwidth}
        \centering
        \includegraphics[width=\textwidth, keepaspectratio]{images/sequence.png}
        \caption{Sequence Diagram}
        \label{fig:sequence_diagram}
    \end{minipage}%
    }
\end{figure}
Figure~\ref{fig:sequence_diagram} shows the sequence of interactions between the User, the Flutter App, the Django Backend, and PostgreSQL. The diagram starts with the user logging in, after which the Flutter app sends an ``Authenticate User'' request to the Django backend. The backend then sends a ``Verify Credentials'' query to PostgreSQL, which responds with User Data. Django then returns a JWT Token to the Flutter app. In the next step, the user completes onboarding, and the Flutter app sends the preferences to Django, which saves the Profile and Preferences to PostgreSQL and confirms success. When the user taps ``Find Roommates'', the Flutter app sends a ``Request Compatible Matches'' call to Django. Django retrieves all candidate profiles from PostgreSQL and then runs a self-call loop labelled ``Calculate Compatibility Scores''. The resulting ``Matching Results'' are returned to the Flutter app. After the user selects a match, the Flutter app sends a ``Request Chat'' to Django, which creates or retrieves a Chat Room from PostgreSQL and returns the Chat Room Details, after which the chat is opened. This diagram covers the full user journey from login to starting a conversation.

Figure~\ref{fig:user_flow_diagram} shows how a user moves through the TRIBAL application from start to finish. The flow begins with launching the app. The first decision is whether the user is an existing user or not. If not, they register an account; if yes, they go straight to Log In. After logging in, there is a second decision: whether onboarding has been completed. If not, the user is taken to Complete Onboarding, and then to Create Profile. If yes, they skip ahead. After the profile is created, the user reaches the Home Dashboard, from which five paths branch out: Find Compatible Roommates, Find Activity Partners, Browse Events, Manage Profile, and Log Out. The first two lead to View Match Details, which further leads to Start Chat. The Browse Events path leads to a decision diamond ``Join or Create Event?'', which splits into Join Event and Create Event. Both event paths and the Start Chat path converge back to Receive Notifications. The Log Out path ends the session. This diagram helped us plan the navigation structure of the app early on so that all screens are connected in a logical way.
