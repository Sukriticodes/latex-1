
\chapter{Methodology}

\section{Process Model}
The team is following the Agile process model for the development of TRIBAL. Agile
was chosen because it allows the project to be broken down into small, manageable parts
called sprints, and each sprint produces a working piece of the application. This is useful
for a minor project because requirements can be refined as the team learns more, and
progress can be shown at regular intervals like this mid-term defense.
The project has been divided into the following sprints:
\begin{itemize}
\item Sprint 1: Project setup, requirement analysis, and UI/UX wireframing in Figma.
\item Sprint 2: Flutter onboarding flow, authentication screens (Login and Signup), and
post-signup profile completion screens.
\item Sprint 3: Django backend setup, PostgreSQL database design, and JWT-based
authentication APIs.
\item Sprint 4: Design and implementation of the roommate compatibility scoring
algorithm (current sprint).
\item Sprint 5: Real-time chat system using Django Channels and WebSockets.
\item Sprint 6: Event creation and joining module.
\item Sprint 7: Integration testing between the Flutter frontend and Django backend.
\item Sprint 8: Final testing, bug fixing, and documentation.
\end{itemize}
At the time of this report, the team has completed Sprints 1 to 4 and has started work on
Sprint 5.

\section{Algorithm}

\subsection{Roommate Compatibility Scoring Algorithm}
The main algorithm used in TRIBAL is the Roommate Compatibility Scoring Algorithm. It compares the onboarding responses of two users and calculates a compatibility score between 0 and 100. The algorithm is rule-based and uses weighted scoring because it is simple, transparent, and suitable for a minor project prototype.
The algorithm takes two user profiles as input: the current user and a candidate roommate. It compares their lifestyle preferences such as cleanliness, budget, sleep schedule, noise level, smoking, guests, interests, study habit, food preference, and drinking habit.
Input
Current user profile and candidate roommate profile containing:
Cleanliness preference
Budget range
Sleep schedule
Noise preference
Smoking preference
Guest preference
Interests
Study habit
Food preference
Drinking preference
Output
A final compatibility score between 0 and 100.


Table 3.1 below shows the weight distribution used in the scoring algorithm.
\begin{table}[H]
\centering
\begin{tabular}{|l|c|}
\hline
\textbf{Factor} &  \textbf{Weight (\%)} \\
\hline
Cleanliness & 15 \\
Budget & 15 \\
Sleep Schedule & 12 \\
Noise Level & 10 \\
Smoking & 10 \\
Guests & 8 \\
Interests & 10 \\
Study Habit & 8 \\
Food Preference & 7 \\
Drinking & 5 \\
\hline
\end{tabular}
\caption{Compatibility Scoring Weights}
\label{table:scoring_weights}
\end{table}
The general formula used to calculate the final compatibility score is:

\begin{equation}
\text{Final Score} = \sum \left(\text{Factor Match Value} \times \text{Factor Weight}\right)
\end{equation}

Where Factor Match Value is a number between 0 and 1 that represents how closely two
users’ answers match for a particular factor, and Factor Weight is the percentage weight
assigned to that factor as shown in Table 3.1.
For factors with a small number of fixed options, such as smoking or sleep schedule, the
match value is calculated using a simple rule: if both users selected the same option, the
match value is 1; if their answers are close but not the same (for ex le, “early riser”
and “no preference”), a partial match value such as 0.5 is given; and if the answers
directly conflict (for example, “smoker” matched with “non-smoker, no smoking
allowed”), the match value is set to 0.
For factors that involve multiple selections, such as interests, the match value is
calculated as the number of common interests divided by the total number of unique
interests selected by both users combined. This gives a percentage overlap between the
two users’ interests.
For numeric factors like budget, the match value decreases as the difference between the
two users’ stated budgets increases. If the difference is within an acceptable range, the
match value stays close to 1. If the difference is very large, the match value moves closer
to 0.


\subsection{Deal Breakers}
Apart from the weighted scoring, the algorithm also includes a deal breaker check. Some lifestyle differences are considered serious enough that they should not simply lower the score a little, but should instead strongly affect or override the final match. For example, if one user is a heavy smoker and the other has explicitly said they cannot live with a smoker at all, this is treated as a deal breaker rather than just a normal mismatch. Table 3.2 lists the deal breaker rules currently used in the algorithm.
\begin{table}[H]
\centering
\begin{tabular}{|p{7.2cm}|p{5.8cm}|}
\hline
\textbf{Condition}  & \textbf{Effect on Score} \\
\hline
Smoking conflict (smoker vs. strictly non-smoking) & Score capped at a low maximum \\
Extreme budget mismatch (outside acceptable range) & Score capped at a low maximum \\
Opposite cleanliness extremes (very messy vs. very strict) & Score significantly reduced \\
\hline
\end{tabular}
\caption{Deal Breaker Rules}
\label{table:deal_breakers}
\end{table}
When a deal breaker condition is detected, the algorithm does not let the final score go above a fixed low ceiling, even if all other factors match well. This makes sure that users are not shown as highly compatible with someone they have a fundamental lifestyle conflict with.

\subsection{Algorithm Steps}
\begin{algorithm}[H]
\caption{Roommate Compatibility Scoring Algorithm}
\label{alg:roommate-matching}
\KwIn{Current user's profile, Candidate roommate profiles}
\KwOut{Ranked list of compatible roommates}

Retrieve the current user's onboarding responses and roommate preferences\;
Fetch all candidate roommate profiles from the database\;
Initialize an empty list named CompatibleCandidates\;

\For{each candidate profile in the candidate list}{
    Set $\text{total\_score} \leftarrow 0$\;
    Compute the compatibility score for each matching criterion:
    \begin{itemize}
        \item Cleanliness preference
        \item Budget preference
        \item Sleep schedule
        \item Noise tolerance
        \item Smoking preference
        \item Guest preference
        \item Shared interests
        \item Study habits
        \item Food preference
        \item Drinking preference
    \end{itemize}\;
    Calculate the total compatibility score by summing all individual criterion scores\;
    Check for deal-breaker conditions (e.g., incompatible smoking preference or budget mismatch). If a deal-breaker exists, reduce the total score or apply a predefined score cap\;
    Round the final compatibility score\;
    Store the candidate profile together with its final compatibility score in CompatibleCandidates\;
}

Sort CompatibleCandidates in descending order based on the final compatibility score\;
Return the sorted list as the ranked roommate recommendations\;
\end{algorithm}

\subsection{Final Score Calculation}
Once all the individual factor match values are calculated and weighted, they are added together to get the final compatibility score. If a deal breaker is triggered, the score is capped according to the deal breaker rule before being returned to the user. The final score is then rounded to the nearest whole number and shown to the user as a percentage match out of 100. This score is what the user sees in the app when browsing potential roommates, helping them quickly understand how compatible they may be with someone before starting a conversation.
This algorithm provides a transparent and explainable matching system. Since it does not depend on machine learning or large datasets, it is practical for the current scope of the TRIBAL minor project.

Figure~\ref{fig:flowchart} shows how the roommate matching algorithm works step by step. The flowchart starts with a ``Start'' node, followed by the step ``User requests roommate matches''. After that, the system fetches the onboarding data of the requesting user from the database, and then fetches all candidate user profiles. For each candidate, the algorithm calculates the individual factor match values and then applies the factor weights. The next step is calculating the total compatibility score, after which there is a decision diamond labelled ``Deal breaker detected?''. If the answer is Yes, the score is capped at the maximum allowed value. If the answer is No, the calculated score is kept as it is. After this, the final score is rounded and all candidates are sorted by their compatibility score. Finally, the ranked list is returned to the Flutter app and the flowchart ends. This diagram helped us make sure that we had thought through all the edge cases in the algorithm before starting to code it.

\end{figure}
Figure~\ref{fig:class_diagram} shows the class diagram for the TRIBAL system. The central class is \texttt{User}, which has four attributes: id, username, email, and password. From the \texttt{User} class, five relationships branch out. The \texttt{User} owns a \texttt{Profile} (one-to-one), which holds gender, age, bio, and location. The \texttt{Profile} further has a one-to-one relationship with a \texttt{Preferences} class, which stores the lifestyle fields used in matching: sleepSchedule, cleanliness, budget, and interests. The \texttt{User} also receives \texttt{Match} objects (one-to-many), where each \texttt{Match} holds a compatibilityScore and a matchedAt timestamp. The \texttt{User} creates or joins \texttt{Event} objects, each of which has a title, description, location, and eventDate. Finally, the \texttt{User} participates in a \texttt{ChatRoom}, which contains many \texttt{Message} objects, each with a content string and a timestamp. The \texttt{User} also has a sends relationship to \texttt{Message} directly. This diagram shows us all the data models we need to build and how they relate to each other.

Figure~\ref{fig:sequence_diagram} shows the sequence of interactions between the User, the Flutter App, the Django Backend, and PostgreSQL. The diagram starts with the user logging in, after which the Flutter app sends an ``Authenticate User'' request to the Django backend. The backend then sends a ``Verify Credentials'' query to PostgreSQL, which responds with User Data. Django then returns a JWT Token to the Flutter app. In the next step, the user completes onboarding, and the Flutter app sends the preferences to Django, which saves the Profile and Preferences to PostgreSQL and confirms success. When the user taps ``Find Roommates'', the Flutter app sends a ``Request Compatible Matches'' call to Django. Django retrieves all candidate profiles from PostgreSQL and then runs a self-call loop labelled ``Calculate Compatibility Scores''. The resulting ``Matching Results'' are returned to the Flutter app. After the user selects a match, the Flutter app sends a ``Request Chat'' to Django, which creates or retrieves a Chat Room from PostgreSQL and returns the Chat Room Details, after which the chat is opened. This diagram covers the full user journey from login to starting a conversation.

Figure~\ref{fig:user_flow_diagram} shows how a user moves through the TRIBAL application from start to finish. The flow begins with launching the app. The first decision is whether the user is an existing user or not. If not, they register an account; if yes, they go straight to Log In. After logging in, there is a second decision: whether onboarding has been completed. If not, the user is taken to Complete Onboarding, and then to Create Profile. If yes, they skip ahead. After the profile is created, the user reaches the Home Dashboard, from which five paths branch out: Find Compatible Roommates, Find Activity Partners, Browse Events, Manage Profile, and Log Out. The first two lead to View Match Details, which further leads to Start Chat. The Browse Events path leads to a decision diamond ``Join or Create Event?'', which splits into Join Event and Create Event. Both event paths and the Start Chat path converge back to Receive Notifications. The Log Out path ends the session.
